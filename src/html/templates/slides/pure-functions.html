Pure Functions

Pure functions have no side effects. That is, pure functions do not modify state or have an observable interaction with the outside world besides returning a value.

  • Wikipedia.org - Side effect

Pure Goodness

Pure functions have many beneficial properties:

  • If the result of a pure function is not used, the function can be removed without affecting other expressions.
  • If a pure function is always called with arguments that have no side-effects, the result of the function for any given set of arguments is always the same.
    • This is sometimes called referential transparency .
    • Makes reasoning about the code much simpler.
    • Makes memoization easy and effective.
  • If there is no data dependency between two pure functions, their order can be reversered or they can be run in parallel without interference.

Only you can prevent side effects!

Pure functions are:
  • Easier to reason about
  • Easier to test
  • Easier to compose
  • Easier to parallelize

Pure Functions

Functional purity is an implementation decision.

  • A lot of bugs are caused by the state being different from what the developer expects at call time.
  • Even so, completely removing side-effects is impractical, if not impossible.
    • A program completely without side-effects sounds pretty useless.
  • A more robust strategy is to isolate the code that mutates state and apply pure transformations to the full application state.
    • This makes it easier to reason about code that doesn't change state.
    • The JavaScript library Redux is a good example of this type of separation of application state and application logic.
